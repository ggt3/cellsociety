8 ###Simulation
9 ####Variety of Grid Location Shapes
10 * We implemented both the square and triangular grid location shapes in our assignment, with each implementation found
11 in the RectangularGridView class and TriangularGridView class, respectively. Each class extends DisplayGrid, which
12 updates the grid that is seen by the user (and is called in the View class). Since the square and triangular shapes of
13 grid locations are similar (with the same number of neighbors), it was easy to use a majority of the same methods to
14 display and update the grid, with the only differing methods relating to drawing the shapes to be displayed (as drawing
15 a Rectangle object and Polygon object with 3 points that flips include different methods). Additionally, we added Toggle
16 Buttons to give the user the ability to choose which shape to display before loading the new simulation.
17 
18 * We did not get a chance to implement the hexagonal grid location shapes, but we discussed how it would be accomplished.
19 There would need to be a method that draws the hexagonal shape like we wrote for rectangular and triangular. Additionally,
20 the rows and columns would visually look a little different, and calculating neighbors would require a method that added only
21 the 6 relevant neighbors for the given shape.

23 ####Variety of Grid Edge Types
24 * We implemented both the finite and toroidal grid edge types, in a similar way to the shapes above. This was done by editing
25 the method that got the cell's neighbors (the finite implementation) to include cells on the opposite side of the grid if it
26 were on the border of the grid so that every cell has a full neighborhood. We changed this in the Grid class by having an
27 abstract method, and extended the Grid class to create subclasses called FiniteGrid and ToroidalGrid.
28 
29 * We did not implement the infinite grid edge type, but a similar method would be used. For an unbounded grid, another subclass
30 called UnboundedGrid would be a subclass of Grid in the model package. It would have full neighborhoods like toroidal, but
31 instead of wrapping around, the grid would just extend outwards. This was difficult to wrap our heads around, as we used arrays
32 and ArrayLists which are ordered to store cell states, and the fact that what was once at position (0,0) could move to a location
33 that has negative indices could cause problems. Therefore, there would have to be a dynamic solution as in how to view the scrolled
34 grid and store the information in cells that are not necessarily visible.

42 ###Configuration
43 ####Error Checking
44 * To implement error checking, we added try and catch methods to the controller. In the loadFile method, we added specific events
45 that would notify the user and stop loading the file if no simulation name were given or invalid state values were given. To check
46 for invalid cell states or out of bounds cells, we added try and catch statements to the the XML parser class. Finally, to ensure that
47 default values were given when no parameters were specified, we first created a resource files that contained default values for each
48 simulation, then we created a method called validate parameters in the controller that went through and filled any empty spots in the
49 attribute map.
50 
51 ####Initial Configuration
52 * We implemented the initial configuration by specific locations and states through an XML file parser (XMLParser class). The file provided
53 the size of the grid, colors to be represented by each state (in a color map), states for each location in the grid, the name of the
54 simulation, and the author in the XML file. The parser interpreted the information, and if something was missing, it threw an exception.
55 Otherwise, it put the information into a Packager to be used by the View and Controller.
56 
57 *We did not implement randomly populating the grid, because sending back the information given by the user in the View to the Controller
58 to send to the Model required going through many different classes and we decided that it made the rest of our code more confusing. However,
59 the implementation would have been that user input in the View (indicated by pushing the toggle button for randomizing the grid) would have
60 created a random grid based off of colors in the ViewPackager, and the controller would translate the colors of the grid into states to send
61 back to the model. Then, the model would have control of the grid to update it for the simulation.
62 
63 ####'Styled' Simulations
64 * We implemented a toggle bar in order to allow the user to style the simulation, which includes choosing the shape of the grid locations,
65 the edge type, and whether there were grid lines in the visualization. The ToggleBox class was implemented in the View. If we were to add special
66 colors/shapes/images for the patch states, we would implement the resource bundle to be able to offer a range of options to be represented in
67 a properties file.

69 ###Visualization
70 * We implemented a graphical representation of the different populations shown in the grid with an XYChart to create a line graph. The ViewPackager
71 passed in information about the different colors and populations of different species in the grid. It updates with every generation in the grid,
72 with the code being in the PopulationGraph class.

74 * We discovered how to use setOnMouseClicked to register when cells were touched, and we would use this capability to let the user change the state
75 of the cell by clicking on the shapes in the representation on the screen. Additional code in the controller would be necessary to update the initial
76 states of the simulations if this capability was used. Similarly, a browser window could pop up and allow users to change parameters of the simulations,
77 and the information would be sent from the View to the Controller in order to reconstruct the simulation. In order to allow users to store the information
78 they used to design the initial states and save them, we would need to construct a free hierarchy of nodes, which is the opposite process of the XMLParser.
